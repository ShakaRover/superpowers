# Superpowers 全自动化改造设计（默认无中断）

## 背景与目标
用户要求把 superpowers 打造成全自动化版本：所有操作默认接受推荐、自动执行、无任何中断。核心目标是在**不削弱原文流程强度**的前提下，实现“默认自动继续”。

## 术语与字段定义
- **确认点**：任何要求“用户同意/批准/审阅后才能继续”的句子或步骤。
- **慢速模式**：用户显式要求“每一步都要确认再继续”，此模式下可允许中断。
- **完成宣称守恒**：即使流程继续执行，也不得宣称“完成/通过”，除非验证与审阅明确通过。
- **覆盖块**：每个技能仅放置一个 AUTO-CONTINUE 模板块，作为全局规则，不在每个确认点重复模板。
- **视觉伴侣**：用于展示原型、图表、对比的浏览器可视化辅助；全自动模式下默认不启用，且不再出现询问语句。

## 范围
- 覆盖所有技能（当前清单）：
  - `brainstorming`
  - `dispatching-parallel-agents`
  - `executing-plans`
  - `finishing-a-development-branch`
  - `receiving-code-review`
  - `requesting-code-review`
  - `subagent-driven-development`
  - `systematic-debugging`
  - `test-driven-development`
  - `using-git-worktrees`
  - `using-superpowers`
  - `verification-before-completion`
  - `writing-plans`
  - `writing-skills`
- 不改变技能的步骤结构与核心语义，仅改变“是否等待用户确认/停顿”的行为。
- **例外条款（审批语义统一改写）：** 所有“确认/审阅/批准”的语义统一改写为“自动通过并继续”，这是本改造允许的语义改变之一（执行策略层面）。
- **例外条款（STOP 语义分类）：** STOP 语义允许按 TERMINAL/RECOVERABLE 分类重新解释，以保持流程强度但消除等待。

**新技能接入规则：**
- 新增或自定义技能必须插入 AUTO-CONTINUE 模板；否则视为未完成全自动改造。
- **接入验收清单：**AUTO-CONTINUE 位置符合规则、确认点替换完成、STOP 标注覆盖率 100%。

## 非目标
- 不改变技能的适用条件与触发规则。
- 不删除任何步骤、检查或流程节点。
- 不默认跳过验证或审阅（仍执行，但不等待用户确认；必需输入缺失不适用于验证/审阅步骤）。

## 设计原则
1. **流程保留**：保留所有原有流程与强制语气（MUST/NO EXCEPTIONS）。
2. **最小改动**：以增补覆盖为主，不重写主体。
3. **无中断**：任何“确认/审阅/是否继续”的步骤自动通过并继续。
4. **可追溯**：自动通过必须写明假设、默认选择与适用范围。
5. **完成宣称守恒**：自动继续不等于自动“完成/通过”，失败仍必须标注未通过。
6. **审批语义例外**：任何“must wait/确认”类强制语气，统一解释为“必须自动继续并记录”，不视为削弱强度。

## 核心方案
为每个技能新增统一的“全自动覆盖块（AUTO-CONTINUE）”，要求：
- 遇到确认点、审批点、审阅点时，**默认自动批准并继续**。
- 仍执行相关动作（如生成文档、派发审阅子代理、运行验证），但**不等待用户响应**。
- 当有风险或不可逆动作时，**仍自动执行**，但必须记录风险提示（慢速模式下此条被覆盖，见优先级规则）。
- 与“验证/完成宣称”冲突时，以“完成宣称守恒”优先：可继续执行，但不得宣称已完成或通过。
- **必需输入缺失处理（优先级）：**
  1) 若存在推荐默认值，则直接使用并记录“使用默认值”。
  2) 若默认值为 `N/A`，则记录“缺失输入-不可执行”，状态为“已跳过（缺失输入）”，并跳过该动作继续。
- **慢速模式优先级：** 一旦进入慢速模式，所有不可逆动作必须等待确认（覆盖“自动执行”规则）。
- **示例：** 当前步骤检测到慢速模式并触发不可逆动作 → 先等待确认，再执行；若已开始执行则记录“无法中止”。

## 组件与职责
### 1) 自动覆盖块（AUTO-CONTINUE）
- 作用：替代“等待确认”的停顿行为。
- 放置位置（优先级从高到低）：
  1. Overview 段落下方
  2. Key Principles 段落上方
  3. 若两者均不存在，放在文档首个二级标题前
  4. 若无二级标题，放在文档首段之后
- 统一模板（每个技能仅放置一处）：
  - `AUTO-CONTINUE: ON`
  - `默认值/推荐答案`（无默认值时写 `N/A`）
  - `推荐原因`（无推荐时写 `N/A`）
  - `假设与适用范围`
  - `风险提示（如有）`
  - `完成宣称约束（失败不等于完成）`

**示例片段（模板式，非具体内容）：**
```
AUTO-CONTINUE: ON
- 默认值/推荐答案：...
- 推荐原因：...
- 假设与适用范围：...
- 风险提示：...
- 完成宣称约束：若验证失败/审阅未通过，只记录失败，不宣称完成
```

### 2) 确认点识别与替换规则
**决策表（可测试）：**
- 若触发排除条件 → 不替换，仅追加“默认继续”。
- 否则若满足语义判定 → 视为确认点并替换。
- 否则若关键词命中 → 视为确认点并替换。
- 否则 → 不替换。

**判定流程（从高到低）：**
1. 语义判定是否为流程 gate（见下方规则）
2. 关键词命中（仅在未触发排除条件时生效）
3. 人工标注（在技能中标注 `CONFIRMATION-POINT:` 行）

**语义判定最小规则（可测试）：**
- 句子包含“需要/必须 + 用户 + 同意/批准/确认/审阅 + 才能继续/才能执行”。
- 句子表达“等待用户反馈/批准后继续”。

**排除条件（优先级最高，避免误判）：**
- 若句子只是**解释性描述**而非流程 gate（例如“用户可以确认一下”），则不替换，仅在句末追加固定话术：“（已默认继续）”。
**混合句式处理：**若句子同时包含建议性措辞与 gate（例如“建议先确认后再继续”），按 gate 处理并替换为默认继续。
**输出规则：**每条只产生一种结果：要么“替换”，要么“追加”，不得同时出现。

**最小例表（可测试）：**
- 正例（应替换）："请确认后继续" / "批准后执行" / "approval gate" / "请审阅再继续" / "需要用户确认才能执行"
- 反例（不替换，仅追加）："用户可以确认一下" / "确认有助于理解" / "建议确认范围" / "确认点说明" / "确认是可选的"

**人工标注格式：**
- 单独一行：`CONFIRMATION-POINT: <原话术>`
- 放置位置：紧贴原确认话术上一行

**识别关键词（命中且未触发排除条件才视为确认点）：**
- “请确认” / “请审阅” / “是否继续” / “等你确认” / “批准后” / “review gate”
- “approval gate” / “ask user to review”

**替换规则：**
- 将确认点话术替换为“已默认接受并继续”（不重复 AUTO-CONTINUE 模板块）。
- 如为“视觉伴侣”询问，改为“默认不启用视觉伴侣并继续（可显式开启）”。

**确认点替换示例（格式）：**
- 列表项：`- 请确认后继续` → `- 已默认接受并继续`
- 标题行：`## 请确认` → `## 已默认接受并继续`
- 内联短语：`完成后请确认` → `完成后已默认接受并继续`

### 3) 停顿点替代话术
- 将“请确认/请审阅/是否继续”改为“已默认接受并继续”。
- 视觉伴侣相关询问语句移除，仅在相关位置追加一句：“已默认不启用视觉伴侣并继续”。

### 4) 审阅与验证自动化
- 规格审阅：自动派发子代理，**不阻塞**主流程；首轮等待 60s，超时触发一次重试（幂等），再等待 60s，仍未返回则记录“未知状态”并继续。
- 验证步骤：自动运行，**不阻塞**主流程；首轮等待 60s，超时触发一次重试（幂等），再等待 60s，仍失败则记录失败并继续。

## 冲突处理优先级
1. **完成宣称守恒**（最高优先级）：验证失败或审阅未通过时，允许继续，但不得宣称完成。
2. **STOP-TERMINAL**：立即终止流程，进入安全终点输出失败状态。
3. **外部平台强制审批不可绕过**：不再尝试执行，记录“被平台阻断”。
4. **慢速模式优先**：进入慢速模式后，不可逆动作必须等待确认。
5. **无中断执行**：除 STOP-TERMINAL、外部平台阻断与慢速模式外继续流程、不等待确认。
6. **原流程强制语气**：保留 MUST/NO EXCEPTIONS，但解释为“必须继续并记录”。


## 执行接口（最小集成点）
- **子代理派发触发点：**当技能要求“派发审阅/子代理”时立即派发。
- **超时计时者：**主代理负责计时（默认 60s），到期触发一次重试。
- **幂等/去重：**重试必须携带幂等键（技能名 + 步骤名 + 动作 ID + 时间窗口 60s；以首次触发时间为窗口起点），避免跨步骤误去重。迟到返回一律丢弃，仅记录“迟到响应”。
- **失败回调：**记录“未通过/未知状态”与原因，继续流程。
- **记录落地：**主代理在会话回复中输出状态摘要（不强制落盘）。

## 数据流/流程流转
1. 触发技能 → 进入流程
2. 遇到确认点 → AUTO-CONTINUE 生效 → 自动批准并继续
3. 有子代理/验证 → 自动执行 → 结果记录 → 继续下一步
4. 异步失败/超时/结果缺失 → 记录为“未通过/未知状态”，继续
5. 最终到达流程终点（以技能自身流程图/终点描述为准；触发下一技能仅表示流程继续，不代表完成宣称）

**统一状态输出格式（用于审阅/验证失败）：**
- `状态：未通过` / `状态：未知（超时/缺失）`
- `原因：...`
- `影响：...`
- `继续策略：已自动继续`

**超时与重试（默认值，适用于所有子代理与验证步骤）：**
- 子代理等待超时：`60s`
- 超时后重试：`1` 次；重试仍超时则标记“未知状态”并继续

**验证步骤超时与重试（默认值）：**
- 验证超时：`60s`
- 超时后重试：`1` 次；重试仍失败则记录失败并继续

**超时与重试优先级：**
- 若技能内已有更严格的超时/阻塞规则，以技能内规则为准。
- 否则使用上述默认值。

**默认降级策略（当后续步骤依赖缺失数据时）：**
1. 使用最近一次可用输出
2. 若无可用输出，使用默认值占位
3. 记录“输入缺失”与“降级来源”（上一次输出/默认值）并说明对下游影响

**责任主体与记录落地：**
- **执行者：**主代理（当前会话）
- **记录者：**主代理在会话回复中记录状态摘要
- **记录位置：**仅在会话回复中，不做强制落盘
- **记录格式：**遵循“统一状态输出格式”
- **迟到响应：**仅追加“迟到响应”备注，不更新已输出状态

## 错误处理与边界
- 子代理失败：状态标记“未通过”；子代理超时：状态标记“未知状态”。
- 验证失败：记录失败细节并继续；不得阻塞流程。
- 重大不可逆动作：仍自动执行，但必须增加“风险提示 + 降级/替代路径”；仅 A 级提供回滚建议。
- **安全/合规边界：**涉及支付、发布、删除生产数据时：
  - 慢速模式：必须等待确认后执行。
  - 全自动模式：允许执行，但必须标注“高风险自动执行”并输出降级路径与责任提示。
  - 若外部平台强制审批不可绕过：不再尝试执行，记录“被平台阻断”，标记未知状态并继续。

## 重大不可逆动作定义与分级
**判定标准（满足任一项即为重大不可逆）：**
- 删除/覆盖不可恢复的数据或文件
- 破坏性系统操作（如重置、卸载、清空）
- 对外部系统产生不可撤回效果（如发布、发送、支付）

**分级与处理：**
- A 级（可回滚）：允许执行，必须给出回滚建议
- B 级（不可回滚）：允许执行，必须给出降级/替代路径与风险提示（不要求回滚建议）

## STOP 规则重定义（等待语义例外）
将 STOP 分为两类，并在技能中明确标注：
- `STOP-TYPE: TERMINAL`（终止型）：终止流程，进入“安全终点”输出失败状态，不再执行后续步骤；不触发任何后续高风险动作。
- `STOP-TYPE: RECOVERABLE`（可恢复型）：必须执行 STOP 对应动作（例如删除/重做），然后自动继续。

**安全边界：**仅当 STOP 对应动作需要执行且为高风险/不可逆时：
- 全自动模式：仍执行，但记录“高风险自动执行”。
- 慢速模式：等待确认后执行。
**默认规则：**未标注的 STOP 视为 `TERMINAL` 并记录警告（迁移期安全兜底）。
**RECOVERABLE 失败处理：**若动作失败/部分完成/不可达，状态标记“未通过”，记录原因并继续后续步骤。
**RECOVERABLE 不可执行处理：**若动作依赖缺失输入/外部审批/环境不可达，则标记“未通过”，记录原因并继续。

## 最小示例
**确认点替换示例：**
- 原话术："请确认后继续" → 替换为："已默认接受并继续"

**STOP 标注示例：**
- `STOP-TYPE: RECOVERABLE`（删除并重做后继续）
- `STOP-TYPE: TERMINAL`（终止流程，输出失败状态）

## 测试与验收
### 验收标准
1. 任何技能流程在**无用户响应**情况下可自动走完。
2. 不出现“请确认/请审阅/是否继续”的等待话术。
3. 保留原有流程步骤与强制语气。
4. 自动通过/默认值/假设与适用范围必须明确记录。
5. 有风险提示时必须包含“风险提示”字段。
6. 验证/审阅失败时不得宣称完成。
7. 超时/失败的状态输出格式一致可检验。

### 重点测试场景
- `brainstorming`：设计分节确认、视觉伴侣询问、规格审阅门槛均不等待。
- `writing-skills`：`STOP-TYPE: RECOVERABLE` 自动执行并继续；`STOP-TYPE: TERMINAL` 进入安全终点并输出失败状态。
- `verification-before-completion`：验证失败不阻断流程，但必须输出“未通过”。

## 风险与权衡
- 风险：错误更容易被放大、误判需求的概率上升。
- 缓解：强制记录假设/默认值/风险提示，便于追溯与人工修正。

## 迁移与兼容
- 默认启用全自动模式。
- 若用户明确要求慢速/人工确认，可在单次任务中显式声明，作用域为“当前任务”。
- 慢速触发语句（示例）：
  - “慢一点，一次问一个问题”
  - “每一步都需要我确认再继续”
  - “暂停并等我确认”
- **优先级规则：**
  - 当检测到慢速触发语句时，立即覆盖当前任务内后续所有技能的 AUTO-CONTINUE。
  - 若同一任务内用户明确说“恢复自动/别停”，则从当前步骤起恢复自动模式。
  - 若同一会话后续出现“自动继续/别停”语句，则恢复自动模式（对后续新任务生效）。
- **检测方式：**主代理基于用户当前消息文本匹配触发语句；不跨会话持久化。
- **任务边界定义：**一次用户请求触发的技能链视为同一任务；用户发送新请求即进入新任务。
- **慢速模式对不可逆动作的行为矩阵：**
  - A 级（可回滚）：等待用户确认后执行
  - B 级（不可回滚）：等待用户确认后执行
- **慢速模式下视觉伴侣：**恢复询问与确认（如用户同意则启用）。
- 恢复默认：任务结束后自动恢复全自动模式。


## 标注迁移策略
- **STOP 推断准则：**若原文包含“终止/停止继续/不得继续”，标注为 `TERMINAL`；若包含“修复后继续/重做后继续”，标注为 `RECOVERABLE`。
- **STOP 标注覆盖要求：** 所有现有 STOP 必须补全 `STOP-TYPE` 标注后才视为完成改造。
- **默认行为（未标注时）：** 视为 `TERMINAL` 并记录“未标注 STOP”的警告（迁移期安全兜底）。
- **验收门槛：** 每个技能需完成 STOP 标注检查清单并记录覆盖率为 100%。
